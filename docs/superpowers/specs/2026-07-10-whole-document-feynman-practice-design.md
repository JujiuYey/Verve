# 整篇文档驱动的费曼练习设计

日期：2026-07-10

状态：已确认

## 1. 背景

当前费曼练习把 Wiki 文档再次拆成多个 `learning_objectives`，然后围绕单个小目标创建会话、生成题目并交给 Examiner 判定 `pass / partial / fail`。

这套流程已经偏离费曼学习法：

- 用户面对的是逐知识点通关，而不是把自己当老师讲清一段完整知识；
- 小目标切断了文章内部的上下文、因果和概念联系；
- Examiner 只看到小目标、题目和当前回答，没有看到整篇原文和完整对话；
- `explained / written / verified` 掌握分级容易把代码示例变成强制通关条件；
- 即使用户已经用通俗语言讲清概念，系统仍可能要求写代码，练习因此退化为死板的学生过关游戏。

## 2. 已确认决策

费曼练习以整篇 Wiki 文档为单位。

- 文件夹是用户已经划分好的知识领域；
- 文件是用户已经组织好的连续知识单元；
- 系统不再使用 AI 把文件二次拆成互相孤立的通关知识点；
- 一次费曼练习 session 直接绑定 `document_id`；
- `learning_objectives` 和 ObjectiveGenerator 不再属于费曼练习主流程；
- Markdown 标题只用于阅读导航和理解文章结构，不转换成逐项验收清单。

“围绕整篇文章讲”不等于逐段背诵。用户要讲清的是文章解决的问题、核心道理、关键关系以及自己的理解。

## 3. 核心原则

费曼练习的核心任务是：

> 假设听众完全不懂，请用自己的语言把这篇文章讲得通俗、连贯、准确。

系统应判断：

1. 初学者能否听懂；
2. 核心概念和因果关系是否连贯；
3. 解释中是否存在关键误解、自相矛盾或逻辑跳跃；
4. 哪个地方值得像真实听众一样自然追问；
5. 用户补充后，整段解释是否变得更清楚。

系统不应判断：

- 是否完成了预设知识点清单；
- 是否写了代码；
- 是否满足固定答案模板；
- 是否达到可以“过关”的形式条件。

Go 代码、例子和反例都是解释工具，不是固定要求。只有当用户的解释涉及具体运行行为，并且仅靠语言无法确认时，系统才建议用一小段代码验证。

## 4. 练习体验

### 4.1 开始练习

用户从一篇 Wiki 文档进入费曼练习。页面保留整篇原文及目录，并邀请用户开始讲解：

> 请把自己当成老师，向一个完全不了解这个主题的人讲清这篇文章。可以从它解决什么问题、核心道理是什么、各部分之间有什么关系开始。

### 4.2 用户讲解

用户可以一次写完，也可以分多轮讲。系统应保留本次 session 的完整对话，不把每轮回答当成相互独立的考试答案。

### 4.3 听众式反馈

原来的 Examiner 应收敛为费曼听众或解释评审者。它首先复述自己听懂了什么，然后指出：

- 已经讲清楚的部分；
- 听众可能跟不上的位置；
- 需要补充的一个关键关系；
- 一个自然的后续问题。

反馈不再以 `pass / partial / fail` 为核心，也不显示“未通过”“重新过关”等游戏化文案。

### 4.4 继续澄清

用户针对追问补充解释。系统结合整篇原文和前序对话继续判断，不重复已经讲清的要求，也不突然切换到另一套题目。

### 4.5 结束练习

练习由用户主动结束，或者在双方已经形成清晰解释后由系统建议收束。结束不是“考试通过”，而是形成：

- 本次解释摘要；
- 已经讲清的关键关系；
- 仍然模糊或值得复习的部分；
- 可沉淀到学习记忆中的证据。

## 5. 上下文与数据流

费曼评审必须由服务端构建上下文，不能继续只信任前端传入的 `prompt`。

整篇文档是练习的业务范围，但不等于每轮都必须把全文注入模型。上下文策略按文档规模选择：

- 短文档直接使用完整 Markdown，优先保证整体关系判断；
- 长文档始终提供标题和完整目录，再使用限定 `document_id` 的 RAG 检索相关片段；
- RAG 命中片段时同时补充相邻 chunk，避免把标题下的连续论述再次切成孤立知识点；
- 不能只使用纯 top-k 检索，因为用户没有提到的关键关系不会自然进入 query；
- 多轮对话保留完整 session 历史，超出预算时压缩旧对话，但不丢失当前解释脉络。

每次评审至少包含：

- 当前文档标题、完整目录，以及全文或文档内 RAG 上下文；
- 当前 session 的全部用户解释和系统追问；
- 用户本轮新增的解释；
- 与当前文档相关的学习记忆；
- 明确的费曼听众评审标准。

服务端应根据 `document_id` 读取最新文档内容，避免前端提交过期或被裁剪的知识上下文。

对于超过模型上下文预算的文档，不允许静默截断后仍声称完成了整篇评审。系统必须使用包含全篇目录、相关片段和相邻上下文的文档内 RAG；检索证据仍不足时，应明确说明当前不能判断，而不是假装完成了整篇评审。

## 6. 数据模型方向

### 6.1 Session

`learning_sessions` 从强制绑定 `objective_id` 改为绑定 `document_id`。会话身份由用户、文档和开始时间确定。

项目仍处于开发阶段，不保留旧 session、objective、exercise 或 memory 数据。实现时直接重建 learning schema，不设计双轨字段、数据回填或兼容读取。

### 6.2 Explanation review

评审结果不再围绕 mastery 等级和通过状态。建议记录：

- `clear_points`：已经讲清的内容；
- `confusing_points`：听众仍可能困惑的内容；
- `misconceptions`：明确的事实或概念错误；
- `follow_up_question`：下一轮唯一的自然追问；
- `explanation_summary`：当前解释的整体摘要；
- `ready_to_wrap_up`：是否适合建议收束，但不代替用户决定结束。

### 6.3 Learning memory

学习记忆从完整解释和多轮澄清中提取证据，不再因为单次 `pass` 就写入“已掌握概念”。适合沉淀的事实包括：

- 用户能够建立的概念关系；
- 用户反复出现的真实误解；
- 经追问后成功澄清的内容；
- 用户习惯使用的有效解释方式。

## 7. 需要退出主流程的旧设计

实现新流程时，应逐步移除以下旧边界：

- ObjectiveGenerator 自动拆分知识点；
- 费曼 session 强制绑定 `objective_id`；
- 按 objective 切换和逐项掌握状态；
- 前端构造固定题目再提交给 Examiner；
- `pass / partial / fail` 通关结果；
- `explained / written / verified` 线性掌握等级；
- 未写代码就不能通过的隐含标准。

如果 `learning_objectives` 在仓库中没有其他独立产品价值，迁移完调用方后应删除整个子系统，而不是把它作为隐藏检查清单保留下来。

## 8. 成功标准

新流程满足以下条件时才算回到费曼练习：

1. 用户从文档直接开始练习，不需要先生成学习小节；
2. 系统评审始终能访问整篇文档和本次完整对话；
3. 用户用通俗语言讲清概念后，不会因为没有 Go 代码而被否定；
4. 系统每次只追问一个真正影响理解的问题；
5. 后续问题承接用户已经说过的话，不重复过关要求；
6. 页面不再用通过、失败、关卡和掌握等级塑造体验；
7. 学习记忆记录的是解释证据和认知变化，而不是机械的判定结果。

## 9. 非目标

- 不把整篇文章变成另一套更大的标准答案；
- 不要求用户覆盖原文每个段落；
- 不把代码编写设为所有 Go 知识的验证条件；
- 不由 AI 重新决定用户已经通过文件夹和文件表达的知识边界；
- 不在本设计中保留 objective 作为隐藏关卡。
