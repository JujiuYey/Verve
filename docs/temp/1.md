你的判断是对的，而且当前实现确实存在三个结构性问题。

**1. Coach 的定位和命名都偏了**

现在的 `LearningCoach` 实际职责是：

- 读取用户学习记录
- 查看 Wiki 文档
- 判断下一步该做什么
- 调用工具或把任务交给具体执行者
- 统一维护用户当前的学习过程

这不是“费曼 Coach”，而是整个系统的总调度者。更准确的名字应该是：

- 代码：`LearningSupervisor` 或 `LearningManager`
- 页面：`学习中心` 或 `学习管理器`
- API：`/learning/supervisor/chat`
- 文件目录：不应该放在 `pages/learning/feynman`

费曼练习只是它管理的一种学习活动，不应该反过来成为总入口的名称。

**2. 这里应该使用真正的 Multi-Agent**

我同意你：单 Agent 靠 Prompt 切换“倾听模式”和“澄清模式”，达不到目标。

原因不是模型无法切模式，而是这些角色有不同的目标、上下文和权限：

- 倾听者不能一边检查你的理解，一边替你回答。
- 教学者需要主动补充知识，不能被限制为只能基于你的表达追问。
- Wiki 编辑者具有写入权限，不能和普通对话 Agent 混在一起。
- 总调度者需要根据对话状态决定下一步找谁，而不是自己承担所有职责。

你文章中的 **Supervisor 模式**正适合这里，Plan-Execute 反而不合适。Plan-Execute 更适合无人参与的复杂任务执行；学习过程需要用户不断参与、卡住、追问和改变方向。

我建议的结构是：

```text
LearningSupervisor
├── FeynmanListener
│   └── 倾听解释、识别含混和误解、提出追问
├── LearningTeacher
│   └── 用户不会时讲解概念、补背景、举例
├── WikiCurator
│   └── 判断文档是否缺失、错误或表达不清，并修改 Wiki
└── LearningJournalAgent
    └── 根据练习过程更新学习日志
```

RAG 不是 Agent，它是这些 Agent 使用的工具。

页面上仍然可以只有一个对话框。用户不需要手动切换 Agent：

```text
用户：“这里我不会，文档也没说清楚”
Supervisor
  → LearningTeacher 解释
  → WikiCurator 检查原文
  → 发现文档确实缺失
  → 请求用户确认修改
  → 修改完成
  → 返回 FeynmanListener
  → 用户继续讲
```

所有子 Agent 完成工作后都把控制权交还 Supervisor，这正是文章里 `supervisor.New(...)` 的预构建范式。

另外，当前两个 Agent 虽然名称不同，但实际上都固定使用 `MiniMax-M3`，并没有真正读取系统里的 Agent 模型配置：[chat.go](/Users/jujiuyey/Projects/Verve/server/infrastructure/llm/chat.go:11)。Multi-Agent 改造时也要把每个子 Agent 的模型配置真正接通。

**3. 学习日志为空是因为根本没有写入逻辑**

这不是你操作不对，是代码缺失。

当前学习日志只有：

- 前端列表页面

但仓库里没有任何地方调用 `Journals.Create`。唯一的写入方法在 [journal.go](/Users/jujiuyey/Projects/Verve/server/app/learning/repository/journal.go:39)，没有调用者。

你完成的四轮讲解目前只写入了：

- `learning_explanation_reviews`

不会写入 `learning_journals`，所以日志必然为空。

目标设计里，`LearningJournalAgent` 应该根据解释、反馈、误解和文档范围生成：

然后按 `user_id + folder_id + date` 合并到当天日志。
