# 01 · 用户与核心场景

> 状态:**待深化**。本篇目前是骨架 + 初步假设,需要后续逐项讨论确认。

## 目标用户画像

`(待深化)` 初步假设几类:

- **自学的开发者**:想系统学一门新语言 / 框架 / 技术主题,不满足于零散搜索和一问一答。
- **带资料想吃透的人**:手上有文档 / 教程 / 论文,想被"带着学完并且学会"。
- **想要被监督的学习者**:自制力一般,需要一个会盯进度、会验证、会安排下一步的私教。

待确认:主力用户到底是谁?是否先锁定"技术 / 编程自学者"(与 MVP 内容范围相关,见 [07](07-决策记录与开放问题.md))。

## 典型场景

`(待深化)` 候选场景:

1. **一句话起步**:用户说"我要学 Rust 的并发",平台直接生成路线、开第一节。
2. **传资料起步**:用户上传一份内部文档 / 一本书的章节,平台基于它拆解学习路线并出题。
3. **每日推进**:用户每天回来,平台知道上次学到哪、今天该练什么。
4. **复习召回**:平台标记的薄弱点到期,主动安排复习。

## 端到端用户旅程

`(待深化)` 初步草图:

```text
注册 / 登录
  → 创建一个学习目标(一句话 或 上传文档)
  → 平台生成学习路线,展示阶段与小目标
  → 进入一节课:诊断 → 讲解 → 练习 → 验证
  → 平台判断是否真学会,写入画像 + 日志
  → 推荐下一步 / 安排复习
  → (次日)回到上次进度继续
```

每一步的界面与交互细节见 [05-功能规格](05-功能规格.md)`(待深化)`。
