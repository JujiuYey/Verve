# 03 · 学习闭环与教学法

> 这是产品的**灵魂和护城河**。技术栈可以换,这一篇的东西换不掉。

## 学习闭环

```text
给出目标(一句话 / 上传文档)
→ 生成路线
→ 开始一节课
→ 做一个小练习
→ 验证结果
→ 写入学习状态(画像 + 日志)
→ 推荐下一步
```

**闭环成立的标准**:用户明显感觉"它知道我学到哪了,也知道我下一步该怎么练"。

## 费曼教学循环(单节课内部)

```text
读取学习状态
→ 选择一个小目标
→ 一个热身诊断问题
→ 短讲解
→ 小练习
→ 一个验证命令 / 动作
→ 反思
→ 生成学习日志草稿
```

教学原则:

- 每次只推进**一个**认知点;
- 先诊断,再讲解;
- 不连续追问;
- 讲完给一个小练习;
- 要求用户用命令 / 输出 / 解释来验证;
- 帮用户把概念讲出来。

> "每次只推进一个认知点"是否设为硬约束,待拍板(见 [07](07-决策记录与开放问题.md) Q7)。

## 怎么判断"真学会"(关键难点)

不能停留在"看过""听过"。需要把掌握程度分级,并要求对应的证据:

| 掌握层级 | 判定证据 |
|---|---|
| 看过 | 仅阅读,无产出 |
| 听过 | 能复述,但不一定理解 |
| 能讲出来 | 用自己的话解释清楚 |
| 能写出来 | 能写出代码 / 答案 |
| 能验证 | 能跑出正确结果 / 通过测试 |

学习监督 agent 的职责就是判定用户处在哪一层,并把结论写进学习画像。

## 验证手段(第一版)

第一版**先不做代码沙箱**,验证从这几类开始:

- 用户解释(讲给 AI 听);
- 用户选择题 / 填空;
- 用户粘贴本地运行结果;
- 用户提交最小代码片段;
- agent 判断输出是否合理。

后续再做:浏览器内代码运行、远程容器 sandbox、自动测试、项目级练习。`(待深化)`

## 真正难的地方(产品壁垒)

难点不在 React,也不在 eino。真正难的是:

- 如何判断用户是否真的学会;
- 如何让 agent 不要空泛鼓励;
- 如何避免一直问问题却不教学;
- 如何维护长期学习状态;
- 如何让多个 agent 协作但不打架;
- 如何让学习路线既个性化又不乱跳;
- 如何让用户感到每天都在推进。

这些做对了,产品才立得住。
